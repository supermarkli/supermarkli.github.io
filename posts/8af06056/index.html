LaTeX 编译原理与指令指南

发表于 2025-12-12 分类于 效率工具 阅读次数： 本文字数： 2.6k 阅读时长 ≈ 5 分钟

写 LaTeX 和用 Word 最大的区别在于：它更像是在写代码。既然是代码，就少不了“编译”这个环节。从一个 .tex 文本文件到一个精美的 PDF，中间到底发生了什么？为什么有那么多不同的编译命令（pdflatex, xelatex, lualatex）？

这篇笔记带你快速厘清 LaTeX 的编译流程，从此告别“盲试”编译器的痛苦。

为什么 LaTeX 需要编译？

在 Word 中，排版引擎是实时运行的。而在 LaTeX 中，你写的是“指令”（内容+格式标记），只有通过编译器处理这些指令，计算出每个字符的位置、断行、分页，最终才能生成文档。

一图速懂：编译流程

LaTeX 的编译往往不是一次完成的，特别是涉及交叉引用（目录、图表索引）和参考文献时。

  • .tex: 你的源代码。
  • .aux: 记录章节、引用等信息的辅助文件。
  • .log: 编译日志，报错时看这里。

常见编译器（Engine）对比

虽然统称 LaTeX，但实际上有多个不同的“引擎”在工作。选择哪一个，主要取决于你的语言需求和字体需求。

1. pdflatex

  • 特点: 老牌、速度快、稳定。
  • 输出: 直接生成 PDF。
  • 缺点: 对 Unicode（尤其是中文）支持较弱，需要繁琐的配置；不支持直接插入 .eps 图片（现代版已部分解决）。
  • 适用场景: 纯英文的数学/计算机论文，或者期刊会议模板指定要求使用。

2. xelatex (推荐)

  • 特点: 基于 Unicode，对中文字体支持极好。可以直接调用操作系统里的字体（如微软雅黑、SimSun）。
  • 输出: 生成 PDF。
  • 优点: 配合 ctex 宏包，是撰写中文文档的首选。
  • 适用场景: 中文文档、简历、需要使用特殊系统字体的设计类文档。
  • 避坑: 如果你在 Linux/WSL 下使用 ctex 宏包配合 xelatex，记得安装中文字体（如 fonts-arphic-uming），或者配置 fontset。

3. lualatex

  • 特点: xelatex 的继任者，集成了 Lua 脚本引擎。
  • 优点: 内存管理更现代化，支持用 Lua 脚本控制排版逻辑，功能最强。
  • 缺点: 编译速度相对较慢。
  • 适用场景: 需要复杂编程逻辑的高级排版，或者 xelatex 无法满足的边缘情况。

常见辅助指令

除了主编译器，你可能还会用到这些：

  • bibtex / biber: 用于处理 .bib 参考文献文件。
    • 流程通常是：latex -> bibtex -> latex -> latex。
  • makeindex: 用于生成索引（Index）。

为什么有时候要编译好几次？

你会发现写了 \label{sec:intro} 和 \ref{sec:intro} 后，第一次编译 PDF 里显示的是 ??。

这是因为： 1. 第一遍编译: 编译器读到 \label，将其页码和编号写入 .aux 文件。读到 \ref 时，因为 .aux 还没写完，所以它不知道对应的值，显示 ??。 2. 第二遍编译: 编译器读取已经生成的 .aux 文件，填入正确的引用编号。

这也是为什么很多 IDE（如 VS Code 的 LaTeX Workshop）会自动执行 "Build Chain"，帮你连续运行多次命令。

懒人神器：latexmk

手动敲 xelatex -> bibtex -> xelatex -> xelatex 这一套连招实在太累了。这时候你就需要 latexmk。

它是一个 Perl 脚本，能自动分析依赖关系。它知道什么时候该运行 bibtex，什么时候该重新编译以修正交叉引用，直到文件完全稳定。

总结

  • 写中文：无脑选 xelatex。
  • 写英文 Paper：看模板要求，通常 pdflatex 够用且快。
  • 报错了：先看 .log 文件，如果提示字体缺失（font not found），检查是否误用了 pdflatex 编译中文，或者系统缺字体。
