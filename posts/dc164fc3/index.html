CUDA 软件栈

发表于 2025-09-19 阅读次数： 本文字数： 5.1k 阅读时长 ≈ 9 分钟

本文尝试用一张图把 CUDA 从“硬件→内核→用户态驱动→工具包→深度学习框架”的完整链路讲清楚，并解释常见的版本不匹配问题与定位思路。即便没有 CUDA 基础，也可以按文中的检查顺序逐步排查。

一、全景图（从硬件到框架的调用链）



graph TD;
  HW["GPU 硬件（SM / CUDA Core；NVLink / PCIe 接口）"] -->|"① 内核驱动（syscall / ioctl）"| KMOD["Linux 内核空间：nvidia.ko、nvidia_uvm.ko、nvidia_drm.ko（示例版本：570.133.20）"];
  KMOD -->|"② 用户态驱动（动态链接）"| UDRV["系统 /usr/lib 或 Conda/lib：libcuda.so.*；libnvidia-ml.so.*；libnvidia-ptxjitcompiler.so.*（版本需与内核驱动完全一致）"];
  UDRV -->|"③ CUDA 工具包（动态链接）"| TOOL["CUDA Toolkit / cuDNN / cuBLAS / cuSPARSE：libcudart.so；libcublas.so；libcudnn.so；libcusparse.so（版本可高于驱动，但需 ≥ 最低要求）"];
  TOOL -->|"④ 框架层（Python import / C++ 扩展）"| FW["PyTorch / TensorFlow / Paddle：pip / conda 安装的包；C++ 扩展在运行时 dlopen 以上 .so"];

用白话解释：

  • GPU 上的 SM（Streaming Multiprocessor）里有很多 CUDA Core，负责并行数值运算；NVLink / PCIe 是与 CPU/主机交换数据的总线。

  • Linux 内核模块 nvidia.ko 管控 GPU 的核心功能；nvidia_uvm.ko 负责统一虚拟内存（Unified Virtual Memory，UVM）；nvidia_drm.ko 关联图形显示栈。它们与硬件直连，版本形如 570.133.20。

  • 用户态驱动库位于系统的 /usr/lib 或 Conda 的 lib，典型如 libcuda.so.*（Driver/Runtime API 入口）、libnvidia-ml.so.*（NVML，管理与监控）、libnvidia-ptxjitcompiler.so.*（PTX JIT）。这些库通过 ioctl 与内核模块对话，所以“用户态驱动版本 = 内核版本”必须完全一致。

  • CUDA Toolkit 是开发者库集合（如 libcudart.so、libcublas.so、libcudnn.so、libcusparse.so 等）。Toolkit 版本可以高于驱动，但不能低于驱动要求的最低兼容版本。

  • 框架层（PyTorch / TensorFlow / Paddle）多为“Python 外壳 + C++ 扩展”。运行时会 dlopen() 上述 .so，受环境变量与路径优先级影响。

二、为什么“内核驱动版本 = 用户态驱动版本”？

NVML、libcuda.so 等用户态驱动库通过 ioctl 与内核模块交互，两端必须使用完全一致的结构体布局、ioctl 号与 ABI 协议。如果版本不匹配（例如 570.172 ↔ 570.133），典型现象是：

  • nvmlInit_v2() 返回 NVML_ERROR_LIB_RM_VERSION_MISMATCH (18)；
  • CUDA Runtime 报 cudaErrorUnknown；
  • PyTorch 内部会 TORCH_CHECK 失败并直接抛出断言错误。

结论：

  • 必须保证“内核驱动模块”与“用户态驱动库（libcuda.so / libnvidia-ml.so 等）”版本严格一致；
  • 上层 “CUDA Toolkit（libcudart.so、libcublas.so、libcudnn.so …）” 可以更高，但需满足对当前驱动的最低兼容要求。

三、PyTorch 如何决定用哪份库？

  1. 解释器路径：which python 决定用系统 Python 还是 Conda Python。

  2. LD_LIBRARY_PATH 顺序：进程启动后会按路径顺序 dlopen 所需 .so；Conda 环境的 …/envs/<name>/lib 往往排在 /usr/lib 前面。

  3. 结果：如果 Conda 自带完整 CUDA Toolkit（含 NVML），就会优先加载它；否则回落到系统 /usr/lib。只要首个被找到的 libnvidia-ml.so.1 与内核驱动版本不一致，就会出现 “Driver/library version mismatch”。

四、概念补充（给没有 CUDA 基础的读者）

  • SM / CUDA Core：SM 是 GPU 的并行计算引擎，内部包含若干 CUDA Core；可以把 CUDA Core 理解为做浮点/整数运算的“工位”。

  • ioctl：用户态程序向内核模块发送带结构体参数的“命令”的方式，要求两端对 ABI（应用二进制接口）理解一致。

  • 驱动 vs. 工具包：libcuda.so/libnvidia-ml.so 属于“用户态驱动”，与内核模块一一对应；libcudart.so/libcublas.so/libcudnn.so 属于 CUDA Toolkit，用于开发与加速库调用。

  • SONAME / ABI：Linux 动态库有 SONAME（如 libnvidia-ml.so.1）。相同 SONAME 表示 ABI 兼容，但并不代表与内核模块就能互通版本；用户态驱动仍需与内核模块严格匹配。

七、FAQ

  • 可以只装 CUDA Toolkit 而不装 NVIDIA 驱动吗？不行。libcuda.so 属于驱动栈，来自 NVIDIA 驱动包；没有它，CUDA 运行时无法与内核模块通信。

  • Toolkit 版本一定要等于驱动版本吗？不需要。只要不低于驱动要求的最低兼容版本即可；过高版本通常也能工作，但建议遵循框架官方的兼容矩阵。

  • 为什么 nvidia-smi 失败时 PyTorch 也会挂？因为二者最终都要初始化 NVML 或 CUDA Driver API，与内核/用户态驱动版本不匹配都会失败。

小结

  • 驱动栈 = “内核模块” + “用户态驱动库” + “Toolkit 库” + “框架”。

  • 第一阶（内核 ↔ 用户态驱动）必须同版本；第二阶（Toolkit ↔ 驱动）只需满足最低兼容。

  • 大多数 PyTorch 断言本质是 NVML 初始化失败，根因常见于“路径优先级 + 版本不配套”。按清单逐步核对，通常即可定位并修复。
